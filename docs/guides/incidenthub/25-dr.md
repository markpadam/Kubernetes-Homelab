# Stage 25 — etcd snapshot & restore

**Exam focus:** CKA — etcd backup, restore, cluster recovery.

**Goal:** snapshot etcd, prove the snapshot is restorable. This is the highest-value CKA topic that people skip because it never comes up in normal ops.

---

## What etcd holds

Every Kubernetes object you can `kubectl get` is in etcd. Deployments, Services, Secrets, ConfigMaps, RBAC, Namespaces, CRDs. Lose etcd, lose the cluster. The kubelet keeps Pods running for a while, but you can't change anything.

Application data (SQL databases, Cosmos, blobs) is **not** in etcd — that data lives in whatever store the app uses. You back those up separately.

## Find etcd

In a kubeadm cluster (and Minikube), etcd runs as a static Pod on the control plane:

```bash
kubectl -n kube-system get pod -l component=etcd -o wide
# etcd-minikube  Running  192.168.49.2

ls /etc/kubernetes/manifests/etcd.yaml      # the static pod manifest
```

In a managed cluster (AKS, EKS, GKE), etcd is hidden from you — the cloud provider takes snapshots automatically. Still important to understand.

## Snapshot

The `etcdctl` CLI talks to etcd directly. Needs the client certs:

```bash
ETCDCTL_API=3 etcdctl snapshot save /tmp/etcd-snapshot.db \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key

# Verify
ETCDCTL_API=3 etcdctl --write-out=table snapshot status /tmp/etcd-snapshot.db
# +----------+----------+------------+------------+
# | HASH     | REVISION | TOTAL KEYS | TOTAL SIZE |
# | 1abc...  | 234567   | 4823       | 18 MB      |
# +----------+----------+------------+------------+
```

Move that file off the node — to S3, blob storage, anywhere durable. The snapshot is what you need; the etcd certs are equally important and live alongside.

## Schedule snapshots

A CronJob on the control plane:

```yaml
apiVersion: batch/v1
kind: CronJob
metadata:
  name: etcd-backup
  namespace: kube-system
spec:
  schedule: "0 */6 * * *"
  jobTemplate:
    spec:
      template:
        spec:
          hostNetwork: true
          restartPolicy: OnFailure
          nodeSelector: { node-role.kubernetes.io/control-plane: "" }
          tolerations:
            - { operator: Exists }
          containers:
            - name: etcd-backup
              image: bitnami/etcd:3
              command: [sh, -c]
              args:
                - |
                  TS=$(date +%F-%H%M)
                  etcdctl snapshot save /backup/etcd-$TS.db \
                    --endpoints=https://127.0.0.1:2379 \
                    --cacert=/pki/ca.crt --cert=/pki/server.crt --key=/pki/server.key
                  # upload to blob storage:
                  # az storage blob upload --container-name etcd-backups --file /backup/etcd-$TS.db
              volumeMounts:
                - { name: pki,    mountPath: /pki,    readOnly: true }
                - { name: backup, mountPath: /backup }
          volumes:
            - name: pki
              hostPath: { path: /etc/kubernetes/pki/etcd }
            - name: backup
              hostPath: { path: /var/backups/etcd }
```

## Restore

Restoring etcd is destructive — it replaces the running cluster's data. Steps for a kubeadm/Minikube control plane:

```bash
# 1. Stop the apiserver and etcd
sudo mv /etc/kubernetes/manifests/etcd.yaml /tmp/
sudo mv /etc/kubernetes/manifests/kube-apiserver.yaml /tmp/

# 2. Move the existing etcd data dir aside
sudo mv /var/lib/etcd /var/lib/etcd.old

# 3. Restore the snapshot
ETCDCTL_API=3 etcdctl snapshot restore /tmp/etcd-snapshot.db \
  --data-dir /var/lib/etcd \
  --name minikube \
  --initial-cluster minikube=https://192.168.49.2:2380 \
  --initial-advertise-peer-urls https://192.168.49.2:2380

# 4. Put the manifests back — kubelet will recreate the static Pods
sudo mv /tmp/etcd.yaml /etc/kubernetes/manifests/
sudo mv /tmp/kube-apiserver.yaml /etc/kubernetes/manifests/

# 5. Wait for the API server
kubectl get nodes
```

The cluster comes back to the state at snapshot time. Anything created since the snapshot is gone. Pods that the *kubelet* was running may still be running but will be reconciled to match the snapshot state.

## Velero — application-aware backups

`etcdctl` snapshots all of etcd at once. Velero is the modern alternative for app-level backups:

- Per-namespace backup.
- PV data backup via CSI snapshots or restic.
- Scheduled restores into different clusters.
- Selective restore — "give me back just incidenthub."

Velero stores backups in object storage (S3, blob). For a multi-cluster DR strategy, Velero beats raw etcd snapshots.

## What you learn

- etcd is the cluster state. Lose it = lose the cluster (Pods may keep running, but you can't deploy).
- `etcdctl snapshot save` is the one-liner. Schedule it. Verify with `snapshot status`.
- Restore is destructive and manual — practise it on a non-prod cluster.
- Application data is separate. Back up SQL, Cosmos, blobs through their own mechanisms.

## Try this (exam-form)

```bash
# All-in-one snapshot one-liner for kubeadm clusters
sudo ETCDCTL_API=3 etcdctl snapshot save snap.db \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key

# Verify a snapshot
sudo etcdctl --write-out=table snapshot status snap.db

# List members (read-only)
sudo ETCDCTL_API=3 etcdctl member list \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key
```
